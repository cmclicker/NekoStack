# @nekostack/provenance

> Generated-artifact lineage. Who produced this file? Which prompt? What model? Which retrieval context? The "trace this artifact back to its source" layer.

## Why this exists

When an LLM generates a file, six months later you need to know:
- What prompt produced it?
- What model + version?
- What was the input / context?
- What was the user's role / intent?
- Has the source content changed since (is this artifact stale)?
- What sources informed it (RAG citations)?

Without `provenance`, the answer is "I have no idea, sorry." With it: every generated artifact carries a typed lineage record.

## Scope

### In scope
- Generated-artifact metadata records.
- Generated-by / modified-by metadata.
- Source-file lineage.
- Prompt-to-output trace.
- RAG citation tracking.
- Stale-artifact detection (source changed; output may be stale).
- Source-to-output mapping.
- Provenance manifest files (sidecar `.provenance.json`).

### Out of scope
- Audit log storage (`audit`).
- Prompts (`prompts`).
- Tools (`tools`).
- Changesets (`changeset` — we link).
- LLM safety enforcement (`governance`).

## How this fits the NekoStack

- **`prompts`** emits provenance when generating content.
- **`tools`** records tool-call provenance.
- **`rag`** records citation provenance.
- **`changeset`** links to the prompt-call that produced the changeset.
- **`validator`** uses our stale-detection to flag outdated generated content.

## Design philosophy

- **Every generated artifact is traceable.** No anonymous AI output.
- **Sidecar manifests.** `foo.ts.provenance.json` next to `foo.ts` if metadata can't fit inline.
- **Stale detection.** If source changes, derived artifacts flagged.
- **Citation chains.** RAG → prompt → output is recorded end-to-end.
