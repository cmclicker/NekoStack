# @nekostack/governance

> Policy rules, lifecycle gates, definition-of-done, forbidden-action policies, LLM behavior constraints. The runtime enforcement layer that `lint` can't reach. Where "what 'acceptable' means" lives as code.

## Why this exists

`lint` enforces static patterns ("no direct `process.env` access"). But many rules are *runtime* and *contextual*:

- "Milestones can't transition to `done` without an attached `review` in `approved` state."
- "Changesets that touch `apps/api/src/auth/**` require an explicit policy waiver."
- "LLM sessions must read the affected file before writing to it."
- "No placeholder values (`'TODO'`, `'lorem ipsum'`) in committed code."
- "Generated files must include a `@generated by:` header citing the prompt or generator."
- "Production deploys require a passing health check + a passing smoke test."

These aren't lintable in the traditional sense — they involve state, history, external systems, and policy configuration that varies per project. `governance` is the package that owns them.

A separate `governance` exists (rather than folding into `lint`) because:
1. **Different lifecycle.** Lint runs at edit/CI time. Governance runs at state-transition time (and is queried by humans + LLMs *before* taking action).
2. **Contextual evaluation.** Governance rules depend on the current state of `path` / `review` / `decision` — they're not pure functions of the file at rest.
3. **Waivers + overrides are first-class.** Lint rules disable cleanly; governance waivers are first-class records with a rationale and an expiry.
4. **LLM constraints live here.** "Must-read-before-write" and "no-placeholders" are exactly the kind of LLM behavior constraints that need explicit, queryable owners.

## Scope

### In scope
- Policy rule DSL (predicate + action + waiver semantics).
- Lifecycle gates (state transitions that require gate approval).
- Definition-of-done templates per artifact type.
- Acceptance criteria templates.
- Forbidden-action / kill-switch declarations.
- LLM behavior constraints (must-read-before-write, no-placeholders, must-cite-source).
- Rule waivers / overrides with rationale + expiry + audit.
- Enforcement profiles (strict / relaxed / per-project).
- Runtime policy evaluation API (`governance.check(action, context)`).
- Gate definitions for `path` (milestone transitions) and `changeset` (apply-time).

### Out of scope
- Static lint rules (those are `lint`).
- Authorization decisions (those are `auth` + `permissions`).
- Compliance frameworks (GDPR/HIPAA/SOC2) — those live in `compliance` with governance referenced for enforcement.
- Per-tenant policy customization — TBD; v1 is project-scoped.
- Approval workflow UI — that's `review` (we provide the gate; review provides the UI).

## Competitors and adjacent tools

| Tool | What they do well | Where they fall short for us |
|---|---|---|
| **OPA (Open Policy Agent)** | Powerful policy-as-code, Rego DSL. | External service, Rego learning curve, heavyweight for solo project. |
| **Cerbos / Oso** | Authorization-shaped policy engines. | Authorization-focused; we need broader runtime gates. |
| **Sentinel (HashiCorp)** | Policy-as-code for Terraform. | Vendor-coupled to HashiCorp products. |
| **Custom if-statements** | Cheap. | Drift, no audit, no waiver workflow, no LLM-readable contract. |
| **lint + CI checks** | Static enforcement. | Static only — can't reason about runtime state, project history, or session context. |
| **AI safety / constitutional approaches** | LLM-focused behavior constraints. | Model-side; we're the application side that constrains LLM actions in our own loop. |

The right framing: **a lightweight, schema-typed, audit-integrated, waiver-aware runtime policy engine that knows about NekoStack state** (path/review/decision/changeset). Closest in spirit to OPA but designed for solo-dev workflows rather than enterprise policy administration.

## How this fits the NekoStack

- **`lint`** is the static counterpart. Together they cover code-time + state-time enforcement.
- **`path`** transitions milestones through gates governed by this package.
- **`review`** is triggered when a gate requires human review.
- **`decision`** records are cited by policies as the rationale.
- **`changeset`** runs gate evaluations before applying edits — especially relevant for LLM-driven edits.
- **`audit`** receives every gate decision and waiver.
- **`compliance`** profiles (GDPR/HIPAA/SOC2) compose into governance enforcement profiles.
- **`prompts`** encodes LLM behavior constraints declared here (must-read-before-write, etc.) into prompts.

## Design philosophy

- **Policy is data.** Rules are typed records, version-controlled, auditable.
- **Waivers are first-class.** Need to violate a policy temporarily? Create a waiver with rationale, expiry, and audit. Don't sneak around the rule.
- **Composition over magic.** Rules compose explicitly via and/or. No hidden priority orderings.
- **Profiles, not flags.** Enforcement profiles (`strict` / `relaxed` / `experimental`) let projects pick their stance.
- **LLM constraints are first-class.** Where rules constrain LLM actions, they're written in a form `prompts` can consume.
- **Deny-by-default for sensitive operations.** Untyped overrides aren't allowed; everything requires an explicit waiver path.

Example rule:

```ts
import { defineRule } from '@nekostack/governance';

export const milestoneDoneRequiresApprovedReview = defineRule({
  id: 'path.milestone.done.requires-review',
  profile: 'strict',
  when: (event, ctx) =>
    event.type === 'milestone.transition' &&
    event.to === 'done' &&
    ctx.review?.state !== 'approved',
  then: () => ({
    deny: true,
    reason: 'A milestone cannot transition to done without an approved review.',
    waiverPath: 'governance waive path.milestone.done.requires-review',
  }),
});
```
